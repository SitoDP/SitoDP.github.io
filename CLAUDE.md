This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Common commands

```bash
npm run dev              # Vite dev server
npm run build            # vue-tsc type-check + vite build (must pass before committing)
npm run preview          # Preview built dist/
npm run type-check       # vue-tsc only, no emit
npm run test             # Vitest watch mode
npm run test:run         # Vitest single run
npm run test:run -- src/components/__tests__/BookingModal.spec.ts   # Single file
npm run sync:products    # Pull Fender Design catalog (requires DEEPL_API_KEY in .env)
```

Always run `npm run build` (not just `type-check`) before committing — `vue-tsc` catches template-level type errors that the editor doesn't.

## Deployment

- **Hosting**: GitHub Pages, served from the `gh-pages` branch (auto-generated by CI).
- **Trigger**: every push to `main` runs `.github/workflows/deploy.yml` → `peaceiris/actions-gh-pages` publishes `dist/`.
- The user pushes manually from their terminal (Claude cannot push due to auth). After committing, tell the user to `git push`.
- A separate cron `.github/workflows/sync-products.yml` runs `sync:products` daily at 06:00 UTC and auto-commits any catalog changes. **This means `main` can advance behind your back** — when push is rejected, resolve with `git pull --rebase -X ours` (prefer remote for JSON files).
- `DEEPL_API_KEY` must be configured as a GitHub repo Secret for the cron's translation step.

## Architecture

### Routing & i18n (single source of truth: `src/composables/useLanguage.ts`)

- Bilingual ES/EN via **path prefix**: `/...` is Spanish, `/en/...` is English. Every Spanish route has a sibling EN route — when adding a route, add **both**.
- The router uses `createWebHistory`, so direct URLs require a SPA fallback. `public/404.html` redirects unknown paths to `/`, and an inline script in `index.html` restores the original path from `sessionStorage`. Don't remove either.
- Components consume language via `useLanguage()`:
  - `lang` — reactive `'es' | 'en'` derived from path
  - `to(esPath)` — converts `/foo` to `/en/foo` when in EN mode (use this for every internal link)
  - `useT('pageKey')` — typed translation lookup against `src/i18n/index.ts`
- Translations are **a single hand-maintained TS file** (`src/i18n/index.ts`), not extracted strings. Add new keys to both `es` and `en` blocks — TypeScript will complain if a key is missing on one side.

### Shop subsystem

- Catalog is **dropshipped from Fender Design's Shopify store** (we are intermediaries; they ship). Source of truth: `src/data/products.json` + `src/data/collections.json`, regenerated by `scripts/sync-products.ts`.
- The sync hits `https://fender-design.com/products.json` and `/collections.json`, normalizes the data, **translates DE→ES with DeepL** (HTML-aware), and writes back. Translations are cached in `src/data/.translation-cache.json` (committed) so re-runs are nearly free.
- Configurable products (price 0 + tag "Configurador") are flagged `configurable: true` during sync. The UI replaces "Add to cart" with "Solicitar presupuesto" (opens `QuoteModal.vue`) for these — they cannot go through the regular checkout because their price depends on customer specs.
- Category hierarchy is **manually curated** in `src/data/category-tree.ts` (Shopify only exposes flat collections). Top-level groups (Hogar / Textiles / Innovaciones) reference Shopify collection handles as subcategories. Update this file when adding/moving categories.
- Cart state lives in `src/composables/useCart.ts` — a plain Vue ref persisted to `localStorage`. **There is no Pinia/Vuex** in this project; composables hold all shared state.
- Stripe checkout (`useCheckout.ts`) POSTs cart contents to a Cloudflare Worker at `VITE_STRIPE_CHECKOUT_URL`. **The Worker is not yet deployed** — the button will throw if the env var is unset. When implementing the Worker, it must create a `Stripe Checkout Session` server-side (secret key never on client) and return `{ url }`.
- Tienda filters live in URL query (`?cat=...&sub=...`) so deep links and browser back work. `ProductoDetalle.vue`'s back link reads `window.history.state.back` to decide between `router.back()` and a fallback navigation.

### Assets

- **Hero videos are hosted on Cloudflare R2** (`https://pub-e19f3243d179441cbc3adda22f8d74fc.r2.dev/<file>.mp4`), not in the repo. `*.mp4` is `.gitignore`d. If you need to add a new video, upload to the R2 bucket first, then reference the public URL.
- Local images in `src/assets/` should be imported in the `<script setup>` block (so Vite hashes them) — relative `url('../assets/...')` works in `<style>` too. Avoid `@/assets/...` aliases in CSS; Vite doesn't resolve them in `<style>` blocks reliably.

### External integrations

- **Forms** (`BookingModal`, `QuoteModal`, contact) post to a Google Apps Script endpoint (`VITE_SCRIPT_URL`) using `mode: 'no-cors'`. The script differentiates submissions by the `type` field (`reserva`, `quote`, etc.). When adding a new form type, also update the Apps Script handler.

## Conventions

- Vue SFCs use `<script setup lang="ts">` with `<style scoped>`. Match the existing component style — kebab-case CSS classes, CSS variables from `src/styles/main.css` (`--color-primary`, `--font-heading`, etc.).
- View components (`src/views/*.vue`) own their layout including hero section and inline i18n object via `computed(() => lang.value === 'en' ? {...} : {...})` for view-specific labels not worth adding to `src/i18n/index.ts`.
- Type-only imports use `import type { ... }`. JSON imports may need `as Type[]` casts because Vite's inferred type is too narrow (e.g., missing `alt` fields).
